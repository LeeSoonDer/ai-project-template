# 02 — 動態模型調度與熔斷守則（DISPATCH PROTOCOL）
# 地位：核心規則。每個 session 開始時必須完整讀取本檔。
# 模型級別：STRONG / MID / SMALL（定義見 CLAUDE.md）。

═══════════════════════════════════════════
第一章：任務交辦三要素
═══════════════════════════════════════════

任何交辦（人→模型、模型→subagent）缺少以下三要素之一，
接收方必須拒絕開工並要求補齊。這是接收方的義務，不是選項。

要素 1 — 目標動機（WHY + WHAT）
  格式：「目標：[做什麼]。動機：[為什麼做，服務於什麼上層目標]。」
  動機欄位的作用：接收方遇到未預見情況時，用動機判斷邊界，
  而不是擅自猜測或擅自擴大範圍。

要素 2 — 驗收條件（DONE-WHEN，必須可二分判定）
  合格範例：「DONE-WHEN: pytest 全綠 + 新增函式有 type hints + 無新 lint error」
  不合格範例：「DONE-WHEN: 程式碼質量良好」（無法二分判定，禁止）
  每個條件必須能回答 PASS 或 FAIL，不存在中間態。

要素 3 — 回報格式（REPORT FORMAT）
  標準回報格式，接收方完工時逐項填寫：
    RESULT: DONE / BLOCKED / PARTIAL
    CHANGED: [動過的檔案清單，一行一檔]
    VERIFIED-BY: [驗證方式與結果]
    DEVIATION: [與任務卡的任何偏差，無則寫 none]
    NEXT: [建議的下一步，或 none]

第二章：升降級熔斷機制
═══════════════════════════════════════════

規則 2.1 — SMALL 模型熔斷（一次即斷）
  SMALL 模型執行中出現任何錯誤（測試失敗、指令報錯、輸出不符驗收條件）：
  → 立即停止。禁止自行重試。禁止自行修改方案。
  → 輸出升級包（格式見規則 2.4），升級至 MID。
  理由：SMALL 模型的重試會在錯誤方向上加深，浪費 token 且污染上下文。

規則 2.2 — MID 模型熔斷（兩次即斷）
  MID 模型在同一個子任務上連續卡關 2 次
  （卡關定義：一次完整的「假設→修改→驗證」循環以失敗告終）：
  → 第 2 次失敗後立即停止。禁止第 3 次嘗試。
  → 輸出升級包，強制升級至 STRONG。
  同一任務總重試上限：2 輪。超過 = 方向錯誤，不是執行問題。

規則 2.2b — STRONG 模型熔斷（升級鏈終點）
  STRONG 在同一子任務連續失敗 2 次：
  → 停止。升級對象是人類使用者。
  → 輸出：「⚠️ ESCALATION TO HUMAN: [升級包] + 我已窮盡的路徑 +
    我認為需要人類提供的資訊/決策是 [具體項目]。」
  禁止 STRONG 無限重試。升級鏈：SMALL → MID → STRONG → 人類。

規則 2.3 — 降級規則（問題解開後立即降級）
  STRONG 模型解開卡點後，禁止繼續執行批量套用。
  必須輸出「解法卡」交還低階模型：
    PATTERN: [問題的本質是什麼]
    FIX: [解法的精確步驟，SMALL 模型可逐字執行]
    APPLY-TO: [需要套用的所有位置清單]
    WATCH-FOR: [套用時可能遇到的變體及對應處理]
  之後批量套用由 SMALL 執行。STRONG 只在 SMALL 再次熔斷時介入。

規則 2.4 — 升級包標準格式（升級時必須攜帶，缺項禁止升級）
    ESCALATION FROM: [SMALL/MID]
    TASK: [原始任務卡原樣複製]
    ATTEMPTS: [每次嘗試的假設 + 動作 + 結果，逐次列出]
    ERROR LOG: [完整錯誤訊息原文，禁止摘要，禁止改寫]
    FILES TOUCHED: [動過的檔案 + 當前狀態]
    MY HYPOTHESIS: [失敗方對根因的猜測，允許猜錯，禁止空白]
  理由：上級模型最貴的成本是重建上下文。完整的失敗軌跡
  讓 STRONG 免於重走已被證偽的路徑。

第三章：獨立驗證機制
═══════════════════════════════════════════

規則 3.1 — 自驗禁令
  任何模型不得宣告自己的產出「已驗證」。
  「我檢查過了，沒問題」不是驗證，是重複同一個盲區。
  產出方只允許宣告：「SELF-CHECK PASSED，等待獨立驗證」。

規則 3.2 — 獨立驗證的執行方式
  驗證必須由「全新上下文（fresh context）的 subagent」執行，
  該 subagent 的輸入只包含：
    (a) 驗收條件（DONE-WHEN 原文）
    (b) 待驗證的產出本體
    (c) 驗證方式指令
  禁止包含：產出方的推理過程、解釋、自我評價。
  理由：看過推理過程的驗證者會被推理帶偏，重現同一個錯誤。

規則 3.3 — 兩種驗證模式
  模式 A — Read-back 驗證（用於文件/設定/規則類產出）：
    subagent 讀產出 → 用自己的話覆述其行為邏輯 →
    對照 DONE-WHEN 逐條輸出 PASS/FAIL。
  模式 B — 執行驗證（用於程式碼類產出）：
    subagent 跑單元測試 / 執行指令 → 回報原始輸出 +
    對照 DONE-WHEN 逐條輸出 PASS/FAIL。

規則 3.4 — 驗證結果的處理
  全部 PASS → 任務關閉，更新 ai/current_state.md。
  任何 FAIL → 回到產出方，FAIL 項目原文進入下一輪任務卡。
  驗證 subagent 與產出方來回超過 2 輪仍有 FAIL → 升級 STRONG 裁決。

第四章：成本紀律
═══════════════════════════════════════════

規則 4.1 — 貴模型禁做清單（STRONG 禁止親自執行）：
  批量讀檔並摘要、格式轉換、跑測試、重複套用已知 PATTERN、
  逐檔搜尋字串、產出後的機械性校對。
  以上一律交辦 SMALL，附完整三要素。

規則 4.2 — 交辦成本判斷式
  「這個子任務需要判斷力，還是只需要服從力？」
  只需要服從力 → SMALL。
  需要局部判斷力 → MID。
  需要跨檔案/跨系統的架構判斷 → STRONG。
