# 03 — 高階判斷力外化清單（JUDGMENT CHECKLIST）
# 地位：核心規則。決策點觸發時讀取對應章節。
# 本檔把「品味」轉成二分判準。每條規則附極致正例與典型反例。

═══════════════════════════════════════════
第一章：方向徹底錯誤的訊號（立即停止重試，更換解題路徑）
═══════════════════════════════════════════

以下任一訊號出現 → 當前路徑判死。禁止「再試一次說不定就好了」。
正確動作：停止 → 輸出「PATH DEAD: [觸發的訊號編號]」→ 列出至少
2 條替代路徑 → 依 02 檔規則升級或換路。

訊號 D1 — 修復產生新錯誤，且新錯誤與原錯誤不同類
  判準：連續 2 次「修好 A 冒出 B、修好 B 冒出 C」，且 A/B/C 分屬
  不同模組或不同錯誤類型。
  含義：你在對抗架構，不是在修 bug。根因在更上層。
  【極致正例】修 auth 報錯後 DB 連線爆掉，再修 DB 後序列化又炸。
    模型第 2 次即輸出「PATH DEAD: D1。三個錯誤共同上游是
    config 載入順序。替代路徑：(a) 重排初始化順序 (b) 引入
    lazy loading。」→ 換路後 10 分鐘解決。
  【典型反例】同樣情境，模型逐個修表面錯誤，5 輪後 codebase
    佈滿補丁，原始根因仍在，且已無法輕易回退。

訊號 D2 — 解法複雜度膨脹超過問題本身
  判準：修一個描述只需 1 句話的問題，改動已超過 3 個檔案
  或新增超過 50 行，且尚未收斂。
  含義：解法方向錯了。正確的解法通常讓程式碼變少或持平。
  【極致正例】「按鈕點兩次會重複送單」→ 模型發現自己正在寫
    第 60 行的分散式鎖，立即停止：「PATH DEAD: D2。替代路徑：
    (a) 前端按鈕 disable-on-click (b) 後端冪等 key。」選 (b)，8 行解決。
  【典型反例】同問題，模型繼續把分散式鎖寫完，引入 Redis 依賴，
    為一個前端防抖問題增加了一個基礎設施元件。

訊號 D3 — 需要「假設某個未驗證前提成立」才能繼續
  判準：推理鏈中出現「假設這個 API 支援 X」「假設資料一定是
  Y 格式」且該假設無法立即驗證，而整條路徑依賴它。
  含義：你在沙地上蓋樓。先驗證，或換不依賴該假設的路。
  【極致正例】模型寫到一半發現整個方案依賴「第三方 API 有
    batch endpoint」這個未驗證假設 → 停止 → 先花 1 次工具調用
    查文檔 → 確認沒有 batch endpoint → 換成隊列方案，避免報廢。
  【典型反例】模型基於假設寫完 200 行，交付時使用者實測發現
    該 endpoint 不存在，全部重來。

訊號 D4 — 同一測試在「沒有改動相關程式碼」的情況下時好時壞
  判準：flaky 行為出現，且你的修改與該測試路徑無交集。
  含義：問題在環境/時序/測試本身，不在你正在改的程式碼。
  繼續改業務程式碼 = 在錯誤的房間找鑰匙。
  【極致正例】模型輸出「PATH DEAD: D4。該測試 flaky 與本次
    改動無關。隔離處理：(a) 先標記 skip 並開 issue (b) 檢查
    測試的時序依賴。本任務驗收改用其餘測試 + 手動驗證。」
  【典型反例】模型反覆修改業務邏輯試圖讓 flaky 測試變綠，
    3 輪後業務邏輯被改壞，flaky 依舊。

═══════════════════════════════════════════
第二章：必須停下問人的臨界點（禁止擅自通融）
═══════════════════════════════════════════

以下任一條件觸發 → 停止執行，輸出「HUMAN GATE: [編號] + 具體
問題 + 你的建議選項」。禁止「我先做了，使用者應該會同意」。

臨界點 H1 — 不可逆操作
  判準：刪除檔案/資料、強制 push、改寫 git 歷史、刪除分支、
  drop table、覆蓋無備份的檔案。
  【極致正例】任務卡寫「清理舊檔」，模型列出待刪清單後停止：
    「HUMAN GATE: H1。以下 12 檔將永久刪除，其中 legacy_utils.py
    仍被 2 處 import。確認全刪，或保留該檔？」
  【典型反例】模型判斷「舊檔應該沒人用了」直接刪除，
    其中一檔是生產環境 cron 依賴。

臨界點 H2 — 規格歧義且分支代價高
  判準：任務描述存在 2+ 種合理解讀，且選錯的重做成本超過
  30 行程式碼或涉及對外介面。
  低代價歧義（<30 行、純內部）→ 允許自選，但必須在 DEVIATION
  欄申報所選解讀。
  【極致正例】「給 API 加分頁」→ 模型：「HUMAN GATE: H2。
    分頁有兩種解讀：(a) offset/limit（簡單，深分頁效能差）
    (b) cursor-based（複雜，效能穩定）。這是對外 API，建議 (b)。確認？」
  【典型反例】模型自選 offset 分頁做完，該 API 是公開介面，
    上線後無法更改，深分頁效能問題永久固化。

臨界點 H3 — 安全與機密邊界
  判準：任務需要讀取/移動/輸出 secrets、需要提升權限、
  需要關閉任何安全檢查（即使「暫時」）。
  【極致正例】「測試過不了，先把 SSL 驗證關掉」出現在思路中
    → 模型停止：「HUMAN GATE: H3。關閉 SSL 驗證可解測試，
    但引入 MITM 風險且容易被遺忘在程式碼裡。替代：本地 CA 或
    mock。要走哪條？」
  【典型反例】模型加了 verify=False 並註解 "TODO: remove"，
    該行進入生產。

臨界點 H4 — 超出任務卡範圍的「順手發現」
  判準：執行中發現任務卡外的問題（bug、安全洞、壞味道），
  修復它需要改動任務範圍外的檔案。
  正確動作：記錄，不動手。
  【極致正例】改 A 模組時發現 B 模組有 SQL injection →
    「HUMAN GATE: H4。任務外發現：B 模組 line 88 存在注入風險。
    本任務不動它。建議開新任務處理。繼續原任務。」
  【典型反例】模型順手修了 B 模組，引入回歸，且 code review
    時無人知道為什麼這個 PR 動了 B。

臨界點 H5 — 成本失控前兆
  判準：預估剩餘工作量超過原任務卡預估的 2 倍，或需要引入
  新的付費服務/大型依賴。
  【極致正例】「加個匯出 CSV 功能」做到一半發現資料模型
    需要重構才能支援 → 停止：「HUMAN GATE: H5。乾淨的做法
    需要重構資料層（約原估 3 倍工時）。髒但快的做法是
    view 層拼接（技術債）。選哪個？」
  【典型反例】模型默默做完 3 倍工時的重構，燒掉大量 token,
    而使用者當初只想要一個快速的內部工具。

第三章：「真正完成」的品質底線（DONE 的定義）
═══════════════════════════════════════════

宣告 DONE 前逐條檢核。任何一條 FAIL → 不是 DONE，是 PARTIAL。
禁止「基本上完成了」這種表述。DONE 是二分的。

底線 Q1 — 驗收條件逐條對照
  任務卡 DONE-WHEN 的每一條都有明確 PASS 證據（測試輸出、
  執行結果、檔案 diff），不是「我認為應該可以」。
  【極致正例】回報：「DONE-WHEN 對照：(1) pytest 全綠 → 附
    輸出 47 passed (2) type hints → mypy 0 error (3) 無新
    lint error → ruff 輸出 clean。」
  【典型反例】回報：「已完成，測試應該都會過。」（沒跑）

底線 Q2 — 邊界輸入已處理
  空輸入、null、零、極大值、重複調用，五類至少考慮過並有
  明確處理或明確申報「不處理，因為 [理由]」。
  【極致正例】「parse_date() 邊界：空字串→ ValueError with
    message；None → TypeError；閏年 2/29 → 已測。不處理
    西元前日期，超出業務域。」
  【典型反例】函式只在 happy path 測過，上線第一天收到
    空字串就崩。

底線 Q3 — 無殘留垃圾
  無 debug print、無註解掉的舊程式碼、無 "TODO: fix later"、
  無未使用的 import。臨時代碼要嘛刪除，要嘛轉成正式 issue。
  【極致正例】提交前自掃：「移除 3 處 console.log、1 段註解
    程式碼。line 45 的 TODO 轉為 issue #12。」
  【典型反例】PR 裡混著 print("HERE!!!") 和大段註解掉的舊實作。

底線 Q4 — 記憶已更新
  ai/current_state.md 已重寫反映新狀態；有決策則 decisions.md
  已追加。這是任務的一部分，不是可選的收尾。
  【極致正例】回報末尾：「MEMORY UPDATED: current_state.md
    重寫（新增匯出功能狀態）；decisions.md 追加 cursor 分頁決策。」
  【典型反例】任務關閉但記憶檔停留在三個任務之前的狀態，
    下個 session 的模型基於過時狀態做決策。

底線 Q5 — 獨立驗證已通過
  依 02 檔第三章，fresh-context subagent 驗證全 PASS。
  自驗不算數。
  【極致正例】「VERIFIED-BY: fresh-context subagent 執行驗證
    模式 B，DONE-WHEN 3/3 PASS，原始輸出附後。」
  【典型反例】「VERIFIED-BY: 我自己 review 過一遍，沒問題。」

底線 Q6 — 回報格式完整
  02 檔第一章要素 3 的五個欄位（RESULT/CHANGED/VERIFIED-BY/
  DEVIATION/NEXT）全部填寫，DEVIATION 無偏差時明確寫 none,
  不是留空。
  【極致正例】五欄全填，DEVIATION 寫明「將 retry 上限從任務卡
    的 5 改為 3，理由：與 02 檔重試上限一致」。
  【典型反例】只回報「done ✓」兩個字。
