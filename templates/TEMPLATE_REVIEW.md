在執行任何動作前，先完成加載程序：
讀取 CLAUDE.md、rules_core/01_WEAK_MODEL_CORRECTIONS.md、
rules_core/03_JUDGMENT_CHECKLIST.md。
你的第一行輸出必須是加載確認行，第二行必須是任務卡。
未輸出這兩行，禁止開始審查。

重要：你是獨立審查者。你收到的輸入不包含產出方的推理過程，
這是刻意設計。你的價值 = 不被原作者的思路帶偏。
用你自己的眼睛從零審視。

TASK: 審查 — [審查什麼：PR / 檔案 / 產出物]
動機：[這次審查要防什麼：合併前把關 / 安全把關 / 驗收]

審查對象：
- [檔案清單 / diff / 產出物本體]

原始驗收條件（DONE-WHEN 原文，逐條驗證）：
- [條目1]
- [條目2]
- [條目3]

審查清單（逐項檢查，逐項輸出結論）：
1. 正確性：邏輯符合驗收條件？邊界輸入（空/null/零/極大/重複）
   有處理？錯誤路徑有處理，無靜默失敗？
2. 回歸風險：改動是否影響範圍外的既有行為？相關測試存在且通過？
3. 安全基線：無 secrets 入碼、無注入風險、使用者輸入有驗證？
4. 殘留檢查：無 debug 輸出、無註解掉的程式碼、無裸 TODO、
   無未使用 import？
5. 品質底線：對照 03 檔 Q1-Q6，逐條 PASS/FAIL。

MUST-NOT：
- 禁止「大致沒問題」式結論。每個審查項必須有明確 PASS/FAIL。
- 禁止對正確的程式碼寫讚美。只輸出問題與風險。
- 禁止順手修復發現的問題。你是審查者，不是修復者。
  發現問題 → 記錄，交還產出方。
- 禁止在證據不足時給 PASS。看不到測試輸出 = FAIL（證據缺失），
  不是 PASS（想必沒問題）。

REPORT FORMAT（完工時逐項填寫）：
RESULT: APPROVED / REJECTED / APPROVED-WITH-CONDITIONS
BLOCKING: [必須修復才能通過的問題，格式：位置 — 問題 — 建議修法]
NON-BLOCKING: [不阻擋但建議處理的問題]
DONE-WHEN 對照: [逐條 PASS/FAIL + 證據]
Q1-Q6 對照: [逐條 PASS/FAIL]
EVIDENCE-GAPS: [證據不足無法判定的項目，無則 none]
